Sur la page d’archive des actualités, certaines erreurs sont également générées par la pagination de WordPress et ne peuvent pas être corrigées directement.

Validation du code HTML de l’intranet

Afin de vérifier la sémantique de l’intranet, j’ai également utilisé l’outil « Total Validator ». Étant donné que j’utilise des frameworks tels qu’Alpine.js ou Livewire, ainsi que des bibliothèques comme Flatpickr, l’outil nous informe de certaines erreurs qui ne sont pas réellement problématiques.

En effet, des attributs comme x-data ou wire:navigate ne sont pas reconnus comme valides par le logiciel, alors qu’ils sont nécessaires au bon fonctionnement du site.

Respect des bonnes pratiques et efforts apportés à la sémantique

Structure sémantique du code

Que ce soit pour le site public ou l’intranet, j’ai respecter les bonnes pratiques en ce qui concerne la sémantique. Premièrement en respectant la structure de base d'un document, en utilisant les balises adaptées :

  • body : Structure du document
  • header : En-tête du document
  • main : Contenu principal du document
  • footer : Pied de page du document

Accessiblité

En ce qui concerne les titres, j'ai utilisé les balises adaptées pour hiérarchiser mes titres, en utilisant les balises h1 à h6. Avoir une structure de titres cohérente est très important pour une navigation optimale dans le site. Mais également pour les lecteurs d'écrans, pour donner l'accès au site à toute personne, même celles portant un handicap.

J'ai aussi fait attention à renseigner des textes alternatifs (alt) sur toutes mes images. Faire attention à ce détail permet aux personnes malvoyantes de bénéficier d’une description de ce qui se trouve sur l'image. Cela leur permet de comprendre le site comme une personne qui ne possède pas ce handicap.

Les labels de formulaires ont également été réfléchis, pour que l'utilisateur puisse comprendre avec quel type de donnée il doit remplir le champ en question. Cela permet aussi d'avoir des formulaires clairs et facilement compréhensibles par l'utilsateur.

Le site public et l’intranet ont tous deux été développés de manière à être navigables entièrement au clavier. Cela améliore l’expérience utilisateur, notamment pour les personnes qui ne peuvent pas utiliser une souris ou qui préfèrent naviguer uniquement à l’aide du clavier.

Métadonnées

Concernant les métadonnées, une attention particulière a été portée à leur mise en place sur l’ensemble des sites. Chaque page dispose d’une balise title. L’attribut lang est également défini pour indiquer aux navigateurs et aux technologies d'assistances la langue a utiliser. Enfin, certaines balises meta ont été ajoutés afin de donner des informations supplémentaires aux navigateurs (auteur, description, etc.).

Microdatas

Les microdatas ont comme objectif principal d’aider les moteurs de recherche à comprendre le contenu des pages. Pour le site public, il est donc primordial de les utiliser sur le site public, pour aider le navigateur à comprendre le contenu du site, mais aussi pour faire monter le site lors dans les résultats de recherche.

En revanche, pour l'intranet, je n'ai pas inséré de microdatas dans le code. L'intranet étant un outil privé et interne à l'asbl, celle-ci n’a pas besoin d'être référencée pour les moteurs de recherche. L'utilisation de microdatas dans ce cas n'est donc pas pertinente.

Gestion des images

Site public

Pour le site public, l'utilisation de l'attribut srcset est indispensable. Cela permet de charger une image plus petite pour les petits écrans et inversément sur les grands écrans.

Étant donné que j’ai réalisé le site en WordPress, j'ai utilisé la fonction wp_get_attachment_image. Cette fonction génère les attributs srcset et sizes automatiquement et donc gère le repsonsive des images automatiquement.

Intranet

Lors de l’upload d'images dans l'intranet, un redimensionnement est effectué directement lors de l’enregistrement des images. Ces différentes tailles sont définies dans une fichier de configuration, ce qui me permet de générer uniquement les tailles qu'il me faut, et les trier dans différents dossiers.

En ce qui concerne l’attribut srcset, je ne l'ai pas utilisé dans l'intranet pour une seule raison. Peu importe la taille de l'écran, l'image ne change pas de taille et donc n’a jamais besoin d’être recharger dans des tailles plus petites. Par contre, pour éviter d'avoir des images de trop grandes tailles, j'utilise les différents images redimensionnés comme expliqué auparavant. Cela me permet d'avoir quand même un contrôle sur mes images sans utiliser l’attribut srcset.
